## ⏳ 将来の検討事項（トリガー条件付き）

### 1. DBインスタンス階層の増強
- **何が問題になるか**: 売買RPC（`buy_shares`/`sell_shares`）の同時実行数は、Supabase の DB接続プール／Compute サイズに依存。爆発的に同時取引が増えると RPC がキューイングし、約定が遅延する。
- **トリガー**: 売買RPCの p95 レイテンシが目に見えて悪化／Supabase ダッシュボードで CPU・接続数が高止まり。
- **対応**: コード変更不要。Supabase の Compute を Small→Medium 以上へ、または Pro プランへ。PgBouncer/Supavisor のプールサイズ調整。
- **コスト感**: 設定だけ。月額が上がる。

### 2. Realtime購読のファンアウト削減
- **何が問題になるか**: ホーム（`MarketGrid`）が**全市場の `outcomes` UPDATE** を購読。同時視聴者×取引頻度ぶんのメッセージが全クライアントに配信され、Realtime の負荷とクライアント描画コストが増える。
- **トリガー**: 同時オンライン視聴者が数百〜に達する／Realtime のメッセージ数が上限に近づく。
- **対応案**:
  - 「表示中タブのみ購読」（`document.visibilityState` で購読を一時停止）。
  - ホームでは購読せず、一覧は5〜10秒ポーリングに切替（詳細ページのみ Realtime）。
  - 購読を「画面内に見えている市場」だけに絞る。
- **場所**: `web/components/MarketGrid.tsx` の `markets-outcomes` チャンネル。

### 3. 一覧の本格ページネーション
- **何が問題になるか**: 現在は `limit(200)` の一括取得。開催中市場が数百を超えると、初期ペイロードと描画コストが重くなる。
- **トリガー**: 開催中（status='open'）市場が常時 150〜200 を超える。
- **対応案**: カーソルベースの無限スクロール（`close_time` カーソル）。`market_sparklines` も可視範囲ぶんだけ取得。
- **場所**: `getMarkets`（`web/lib/queries.ts`）＋ `MarketGrid`。

### 4. キャッシュの能動的バスト（任意・精度向上）
- **現状**: 新規市場の出現や解決は最大10秒の遅延で反映（TTL頼み）。実用上は問題なし。
- **もし即時反映したくなったら**: 管理操作後に `revalidateTag("markets")` を呼ぶ Route Handler を用意し、Edge Function（generate/resolve）から叩く。
- **優先度**: 低。TTLで十分なため、要望が出るまで不要。

## 監視のヒント（増強判断の材料）
- Supabase ダッシュボード: Database の CPU / 接続数 / 遅いクエリ（pg_stat_statements）。
- Vercel: 関数の実行回数・実行時間・エラー率。
- 「売買RPCの遅延」「一覧表示の遅延」「Realtime詰まり」のどれが先に出るかで、上の 1〜3 の着手順を決める。
