## 0. 結論サマリ（先に全体像）

- **何を作るか**: Polymarket型の体験（価格＝確率がトレードで動く）を、**換金不可ポイント**で再現した遊び場。報酬は称号・ランキングのみ（賞品ゼロ）。マネタイズはBETの外側（コスメ・広告・有料情報）。
- **技術の心臓部**: Supabase/Postgres の `security definer` RPC 群（LMSR価格・台帳・取引・解決）。ブロックチェーン不要、全状態はPostgres。
- **フロント**: Next.js (App Router) + TypeScript + Tailwind + Recharts + Supabase Realtime。
- **設計を貫く絶対条件**: 換金不可／有償発行禁止／譲渡禁止／賞品ゼロ。これらは「コードに存在してはならない機能」であり、**不在をテストで担保**する。
- **MVPの定義**: SPEC-01→02→03→04→05 まで実装した時点で「ベットが回り、解決で償還が走る」動くプロダクト。06以降は体験強化とマネタイズ。

2. キーは `.env.local`（gitignore済）にのみ保持。サービスロールキーはサーバー側（Edge Function / 管理処理）でのみ使用し、フロントには絶対に出さない。
3. もし既にGitHubへpush済みなら、Supabaseダッシュボードで**サービスロールキーをローテーション**する。
4. anonキーはフロント公開可（RLS前提）だが、これも `.env.local` 経由で読む。

### Phase 1: 認証・本人性（SPEC-01）— ⏸ LINEログイン採用・後回し
> **方針変更**: 認証方式は **LINEログイン**。本フェーズは後工程で実装する（MVP開発は §開発用スタブ認証で先行）。
> SPEC-01のメールOTP記述は LINEログインに読み替える。「検証済み連絡先＝LINE認証済みアカウント」「1メール=1アカウント」→「1 LINE userId = 1アカウント」。

技術的含み（重要）:
- **Supabase Auth は LINE をネイティブのOAuthプロバイダとして持たない。** 以下のいずれかで橋渡しが必要:
  - (A) LINE Login (OAuth2) → 自前で検証 → Supabase の **カスタムJWT / Sign in with ID token** でセッション発行（推奨候補）
  - (B) LINE Login → Edge Function でユーザー作成し、Supabase Admin API でセッション付与
  - いずれも `auth.uid()` ベースのRLS・RPC設計（SPEC-02〜08）はそのまま流用できる。橋渡し層だけ差し替わる。
- LINE userId は一意 → 1人1ウォレットのdedupに利用。`profiles` に `line_user_id`（unique）を保持。

後で実装するタスク:
- [ ] LINEログイン橋渡し層（上記A/Bのどちらか確定 → 実装）
- [ ] `profiles` テーブル（`line_user_id` unique 含む）＋RLS
- [ ] `admin_users` テーブル＋管理者判定ヘルパ（※これは早めでもよい。Phase 7が依存）
- [ ] オンボーディングUI（LINE認証 → 表示名 → `complete_signup`）
- [ ] `complete_signup()` RPC（`grant_signup_bonus` を呼ぶ）
- **完了基準**: LINE未認証に初期付与されない／二重付与されない／1 LINE userId = 1ウォレット

#### 開発用スタブ認証（MVP先行のため）
LINEログイン実装までの間、開発を止めないために:
- ローカル/ステージングは Supabase Auth の通常メールサインイン（または匿名認証）で `auth.uid()` を確保。
- RLS・RPCは `auth.uid()` のみに依存させ、**認証プロバイダに非依存**に保つ → 後からLINEへ無痛で差替え。

## 4. 法的不変条件＝ガードレール（全フェーズ横断）

これらは方針ではなく「**コードに存在してはならない機能のリスト**」。CIで常時検証する。

1. 残高 = 当該ユーザーの台帳delta総和（常時一致・監査可能）
2. ポイント入手は無償3種のみ（signup / daily / redeem＋refund）。換金・購入・譲渡RPCは存在しない
3. 残高・株・q の書き込みは `security definer` RPC経由のみ（RLSで直書き拒否）
4. 取引は対象marketを行ロックしてから計算・更新（レース防止）
5. 終端状態（resolved/void）は不可逆
6. 端数はシステム有利（買い切上げ・売り切捨て）でポイント総量が膨張しない
7. 1人1ウォレット（検証済み連絡先を必須）
8. 決済はentitlementsのみ更新。`wallets.balance` を増やす決済コードは存在しない

→ **Phase 8 §2 の「不在テスト」を Phase 0〜1 の早い段階で骨子だけ先に置く**ことを推奨。
　以後の全実装がこの境界を越えないガードレールになる。
