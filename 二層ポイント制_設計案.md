# 二層ポイント制 設計案（ミライマ型）

ミライマ（株式会社Masentic）の調査を踏まえた、D-market への二層ポイント制導入の**設計提案**。
これは検討用のたたき台であり、実装前に仁さんの承認を得る。

作成: 2026-06-19 ／ ステータス: **提案（未実装）**

---

## 1. なぜ二層にするのか（法的メリット）

現状のD-marketは「単一残高（ポイント）」。予想で賭けるのも、景品に換わるのも同じポイント。
景品交換を始めると、この単一ポイントが「賭けの対象＝財産的価値」と見られるリスクが残る。

ミライマは**2種類**に分離している:

| 種別 | 役割 | 性質 |
|---|---|---|
| **参加ポイント** | 予想（売買）に使う投票チケット | 価値ゼロ・換金不可・景品に直接交換できない |
| **賞品ポイント** | 景品交換専用 | 換金不可・譲渡不可・有効期限90日 |

**狙い**: 「賭ける行為」は価値ゼロの参加ポイントだけで完結させ、景品に触れるのは賞品ポイントのみ。
→ 賭けの対象に財産的価値が無いことがより明確になり、賭博性をさらに薄められる。

## 2. D-market にどう当てはめるか（推奨案）

D-market は LMSR の自動マーケットメイク台帳を持つ。これを壊さない最小設計を推奨する。

### 推奨: 「市場は参加ポイント完結／賞品ポイントは別レイヤーの報酬」

- **市場の売買・解決は、すべて現在のポイント（＝参加ポイントに改称）で完結。** LMSR・取引RPC・台帳はほぼそのまま。
- **賞品ポイント**は、市場とは独立した**報酬通貨**として新設。以下のルールで付与（案）:
  - (A) **的中報酬**: 市場が解決し、的中した時に少量を付与（例: 払い戻しの n%）。
  - (B) **ランキング報酬**: 週次/シーズンのランキング上位に付与。
  - (C) いずれか/併用は運用で選択。
- **賞品ポイント → 景品**は確定交換（必要pt数で交換、抽選なし）。

この形なら、**既存のLMSR・売買・解決ロジックは無改修に近い**（通貨名の読み替えのみ）。
新規実装は「賞品ポイントの台帳・付与・景品交換」の一層だけで済む。

```
[無償付与] ─→ 参加ポイント ──(売買/LMSR)──→ 市場ポジション
                                              │ 解決（的中）
                                              ▼
                              参加ポイント払い戻し ＋ 賞品ポイント付与(報酬)
                                              │
                              賞品ポイント ──(確定交換)──→ 景品
```

## 3. データモデル案

最小変更で済むよう、**新テーブル追加**で対応（既存 wallets / point_ledger は参加ポイント用としてそのまま使用）。

```sql
-- 賞品ポイント残高（1ユーザー1行）
create table prize_wallets (
  user_id    uuid primary key references auth.users(id) on delete cascade,
  balance    bigint not null default 0 check (balance >= 0),
  created_at timestamptz not null default now()
);

-- 賞品ポイントの不変台帳（付与/交換/失効を記録）
create table prize_ledger (
  id            bigint generated always as identity primary key,
  user_id       uuid not null references auth.users(id),
  delta         bigint not null,                 -- 正=付与 / 負=交換・失効
  reason        text not null,                   -- win_reward|rank_reward|redeem|expire
  market_id     uuid references markets(id),
  redemption_id uuid,                            -- 景品交換時
  expires_at    timestamptz,                     -- 付与分の有効期限（90日）
  balance_after bigint not null,
  created_at    timestamptz not null default now()
);

-- 景品マスタ
create table prizes (
  id          uuid primary key default gen_random_uuid(),
  name        text not null,
  description text,
  image_url   text,
  cost_points bigint not null check (cost_points > 0),  -- 必要賞品pt
  stock       int,                                       -- null=無制限
  is_active   boolean not null default true,
  created_at  timestamptz not null default now()
);

-- 景品交換申込
create table prize_redemptions (
  id          uuid primary key default gen_random_uuid(),
  user_id     uuid not null references auth.users(id),
  prize_id    uuid not null references prizes(id),
  cost_points bigint not null,
  status      text not null default 'requested',  -- requested|shipped|cancelled
  shipping    jsonb,                               -- 配送先（暗号化推奨）
  created_at  timestamptz not null default now()
);
```

- **絶対不変条件（absence test の拡張）**: `prize_wallets.balance` は `grant_prize_points` / `redeem_prize` / `expire_prize_points` の3関数を経由しなければ変動しない（現行の台帳ガードと同思想）。
- 既存の「残高 == Σ台帳」テストを賞品ポイントにも適用。

## 4. 必要なRPC（SECURITY DEFINER）

| 関数 | 役割 |
|---|---|
| `grant_prize_points(user, amount, reason, market_id, expires_at)` | 賞品pt付与（的中/ランキング報酬）。台帳1行＋残高加算 |
| `redeem_prize(prize_id, shipping)` | 確定交換。在庫・残高・有効期限を検証し、残高減算＋申込作成 |
| `expire_prize_points()` | 90日超過分を失効（cron で日次） |
| 既存 `resolve_market` に追記 | 解決時に的中者へ `grant_prize_points` を呼ぶ（案A採用時） |

## 5. 既存ロジックへの影響（最小）

| 対象 | 影響 |
|---|---|
| LMSR（lmsr.ts / 0002） | **なし**（参加ポイントで完結） |
| buy/sell_shares（0004） | **なし**（残高=参加pt） |
| resolve_market（0005） | 軽微（的中者への賞品pt付与を1行追加・案A時） |
| 既存テスト | 参加pt側は不変。賞品pt用テストを新規追加 |
| フロント | 残高表示が2種に。景品交換ページ・賞品pt履歴を新設 |

## 6. UX への影響

- ヘッダー残高: 「ポイント（参加）」＋「賞品pt」の2表示。
- マイページ: 賞品pt残高・有効期限・履歴。
- **景品ページ**（新規）: 景品一覧→必要pt→交換申込→配送先入力。
- 用語の見直し: 現行「ポイント」を「参加ポイント」へ改称（または現状名を維持し賞品ptを追加）。命名はTBD。

## 7. 段階導入案

1. **Phase A（DBのみ）**: prize_wallets/prize_ledger/prizes/redemptions ＋ RPC ＋ テスト。表に出さない。
2. **Phase B（付与開始）**: resolve_market から的中報酬付与を有効化。マイページに賞品pt表示。
3. **Phase C（景品公開）**: 景品マスタ登録＋景品ページ公開＋確定交換＋特商法表記整備。
4. **Phase D（運用）**: 失効cron・在庫管理・配送オペ。

## 8. リスク・検討事項

- **賞品ptの付与レート設計**: インフレ（景品コスト割れ）しないよう、付与量と景品コストのバランスを運用パラメータ化（platform_settings 拡張）。
- **景品表示法**: 景品の最高額・総額規制。確定交換でも「取引に付随する景品類」と見なされ得る設計か要確認 → **弁護士レビュー必須**。
- **配送個人情報**: prize_redemptions.shipping は要暗号化・最小保持。
- **本人確認**: 高額景品時の本人確認・不正（多重アカウント）対策を強化。
- **命名**: 「参加ポイント／賞品ポイント」かD-market独自名か。
