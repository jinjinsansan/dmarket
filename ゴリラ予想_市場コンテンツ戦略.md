# ゴリラ予想 市場コンテンツ戦略

> 「Polymarketの一部だけ残し、残りは日本人初心者が毎日遊べる身近で変わった市場」で差別化する市場設計。
> 作成: 2026-06-29 ／ 前提: 二層ポイント（参加pt換金不可）・LMSR・解決アダプタ（既存）。
> 関連: `ゴリラ予想_リニューアル計画_2026-06-29.md` / SPEC-03(解決) / SPEC-04(供給)

---

## 0. 確定した方針（2026-06-29 仁さん判断）
1. **v1はYES/NO（binary）のみ**。複数選択（A/B/C/D）は後日。当面は「Aが優勝する？YES/NO」を並べて擬似表現。
2. **デイリー超短期市場を目玉**（朝出して夕方〜翌日解決。毎日触る理由＝習慣化、デイリーボーナスと相乗）。
3. **天気カテゴリを初心者の主力**として作り込む（気象庁データで客観解決）。

## 1. 基本思想：残す／作るの切り分け
- **残す（Poly由来・自動解決の強み）**: FX・ビットコイン・誰もが知る政治ニュース・有名スポーツ。`generate-markets` のPolyミラーを **tag_id厳選＋カテゴリ別 poly_max縮小** でノイズ除去。
- **作る（ゴリラ予想オリジナル＝売り）**: 天気・エンタメ/芸能・身近トレンド・ネタ枠。どこの予測市場にもない身近さ。
- **ユーザー作成市場**: 公式が作りきれない「変わった市場」をユーザーが供給（審査制＋管理者解決）。良問が集まるほど差別化が自己増殖。

## 2. カテゴリ設計

| カテゴリ(slug案) | 供給元 | 解決方法 | 例 |
|---|---|---|---|
| FX (`fx`) | Polyミラー＋自前 | price_threshold（自動） | ドル円 週末◯円超え |
| 暗号資産 (`crypto`) | Polyミラー＋自前 | price_threshold（自動） | BTC 年内◯万ドル |
| 政治 (`news`) | Polyミラー厳選 | poly突合／管理者 | 内閣支持率◯%超え |
| スポーツ (`sports`) | Polyミラー厳選 | 管理者（feed確定後自動） | 阪神優勝・W杯 |
| **天気 (`weather`)** ★主力 | 自前テンプレ | **AMeDAS観測＝客観自動** | 今週末の東京は晴れ／最高気温◯℃超え |
| **エンタメ (`ent`)** ★売り | 管理者＋ユーザー | 管理者（公式発表） | 流行語大賞・M-1・映画興収 |
| **トレンド (`trend`)** | 管理者＋ユーザー | 管理者（公開情報） | ガソリン価格・視聴率・SNSトレンド |
| **ネタ (`neta`)** ★ブランド | 管理者（ゴリラの日替わり） | 管理者（客観のみ） | 🦍今日のお題 |
| デイリー (`daily`) ★目玉 | 自前テンプレ（毎朝生成） | price_threshold/天気（自動） | 今日の日経は上がる？ |

実装: `categories` に上記を upsert。`category_feed_settings` で Poly系は `poly_tag_ids`＋小さめ`poly_max`、オリジナル系は `poly_max=0`＋`template_enabled`。

## 3. 天気エンジン（★主力・客観解決の要）

### 3.1 重要原則：予報 ≠ 結果
- **seed/表示**: 気象庁「予報」JSON（`https://www.jma.go.jp/bosai/forecast/data/forecast/{areaCode}.json`、例 東京=130000）。`weathers`/`weatherCodes`/`pops`/`temps`/`tempsMax` を使い、初期確率シードや市場文に利用。
- **解決（結果確定）**: 予報ではなく **AMeDAS の観測実績** を使う（`https://www.jma.go.jp/bosai/amedas/data/point/{stationId}/{yyyymmdd}_{hh}.json`、東京=44132）。当日の降水量合計・最高/最低気温を集計して判定。**キー不要・無料・客観**。

### 3.2 resolution_binding（weather・新規 kind）案
```json
{
  "kind": "weather",
  "station": "44132",            // AMeDAS地点（東京）
  "date": "2026-07-05",          // 評価対象のJST日
  "metric": "precip|temp_max|temp_min",
  "operator": ">=|>|<=|<",
  "threshold": 30,               // temp系=℃ / precip系=mm（"雨が降る"は precip > 0）
  "yes_if_true": true
}
```
- 判定可能になるのは **対象日終了後（翌日01:00 JST以降など）**。それ以前は `pending`（既存規約：未確定は推測せず pending）。
- 観測の取得・形が想定外なら `pending`/`error` で**管理者解決にフォールバック**（fail-safe）。
- 解決アダプタは `_shared/adapters.ts` の `resolveBinding` に `case "weather"` を追加して実装。

### 3.3 天気市場の例
- 今週末（土）の東京は晴れる？（weatherCode/観測で判定）
- 明日の東京の最高気温は30℃超え？（temp_max ≥ 30）
- 今日、東京で雨が降る？（precip > 0）← デイリー枠と相性◎

## 4. デイリー超短期市場（★目玉）
- **生成**: 毎朝（cron）テンプレから当日市場を自動生成（`market_templates.schedule_cron`）。例「今日の日経平均は前日より上がる？」「今日の東京は雨？」。
- **解決**: 当日夕方〜翌朝に自動（日経=price_threshold／天気=weather）。
- **体験**: 朝に出て当日中に決着 → 毎日来る理由。デイリーボーナス受領と同じ動線に置く。
- 実装: `generate-markets` の「自前テンプレ生成（§4a・現状TODO）」を有効化し、`schedule_cron` 評価でデイリーテンプレを走らせる。

## 5. 解決の鉄則（全テンプレ＆ユーザー作成審査の基準）
> **「公開情報で・誰が見ても明らかに・客観的に」確定できる問いだけ採用。**
- ❌ 主観／あいまい（「面白いか」「人気が出るか」等の感想系）
- ❌ 私生活・名誉毀損リスク（芸能人の結婚/熱愛の有無予想など）→ 「公式発表があるか」までに限定
- ❌ 解決ソースが不明な問い
- ✅ 価格・気象観測・公式発表・公的統計・スコアなど一次ソースで確定できる問い

## 6. ユーザー作成市場（審査制＋管理者解決）
- 審査通過ユーザーのみ作成可（リニューアル計画⑥）。審査基準＝§5の鉄則＋不適切テーマ排除。
- **結果確定は管理者/オラクル**（作成者は解決しない＝不正・係争防止）。
- 作成者のテラ銭＝**換金不可の参加ポイント**（賭博非該当の生命線）。

## 8. 実装インクリメント（着手順）
1. **カテゴリ再編 migration**: `categories` upsert＋`category_feed_settings`（Poly系tag_id厳選・オリジナル系 poly_max=0）。※Gammaのtag_id実値は要取得（/admin/settingsで調整可）。
2. ✅ **天気アダプタ（実装済み 2026-06-29）**: `_shared/adapters.ts` に `weather` kind を追加（AMeDAS観測実績で自動解決・キー不要）。実データ（東京2026-06-28）で集計検証済み＝最高23.5℃/最低19.8℃/降水7.0mm。配線済み（`markets.resolution_binding`→`create_admin_market`→`resolve-markets` cron）。**残**: ①リモートへ Edge Function デプロイ ②天気 `market_templates` 追加 ③`/admin/create` で binding を入力できるUI（or テンプレ生成）④AMeDAS地点コード表。
3. ✅ **デイリー生成＋天気テンプレ（実装済み 2026-06-29）**: `generate-markets` の §4a テンプレ生成を実装（`params_source.offsets` の各JST日について `create_market_internal` で生成・`external_ref=tmpl:{id}:{yyyymmdd}` で冪等）。migration `0027_weather_templates.sql`＝天気カテゴリ＋feed設定（poly_max=0・template有効）＋デイリーテンプレ2件（東京の雨 precip>0／真夏日 temp_max≥30、offsets[0,1]＝今日と明日）。ローカルPGで適用・冪等を検証済み。**残**: ①リモートへ migrations push（`npx supabase db push`）②Edge Function デプロイ（generate-markets / resolve-markets ＝ _shared同梱）。
4. **Polyミラー厳選**: keep-4のtag_idを設定し`poly_max`を縮小。オリジナル系は0。
5. スターターセットを `/admin/create` で投入（手動解決系）。
