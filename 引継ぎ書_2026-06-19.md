## 0. 一言で言うと

**D-market**（ディーマーケット）= Polymarket型UXを、**換金不可・無償発行ポイント**で再現した日本語の予測市場。
**賭博非該当**を設計の核に据える（ポイントは無償発行＝ユーザーは自己の財産を賭けない）。
貯めたポイントは将来**景品と確定交換**できる（ミライマ型の二層ポイント制を導入中）。

## 1. 技術スタック

| 層 | 技術 |
|---|---|
| フロント | Next.js 16.2.9 (App Router) / React 19 / TypeScript / Tailwind v4(@theme) / Recharts |

### ローカルDBテスト（Docker不要・PG16バイナリ使用）
```bash

PG_BIN="/c/Program Files/PostgreSQL/16/bin" bash supabase/tests/run_local.sh
# "ALL TESTS PASSED" を確認。0009_cron はローカル非適用（pg_cron/pg_net必須のため）。
# 新マイグレーションは run_local.sh の migration ループとテスト行に追記すること。
```

### マイグレーションの本番適用（重要）
- **コード変更（web/）はpushで自動デプロイされるが、SQLは手動適用が必要。**
- 手順: Supabaseダッシュボード → SQL Editor → 該当マイグレーションファイルを**VSCodeで開いてコピペ**して実行。
- ⚠️ **チャット経由でSQLを貼らない**（`--`がem-dashに化ける等の事故が頻発）。必ずファイルから。
- DDLの正常応答は「Success. No rows returned」。

### デプロイ（Vercel）
- `git push` で main に上げると Vercel が自動ビルド＆デプロイ。数十秒で反映。
- 仁さんが自分で `! git push` する場合もあるが、これまでは Claude がpushしてきた。

## 5. 二層ポイント制（現在ここを開発中）★最重要

ミライマ（株式会社Masentic）調査を踏まえ導入決定。設計詳細は `二層ポイント制_設計案.md`。

### 確定した仕様（仁さんの判断）
1. **参加ポイント**（既存 wallets.balance）= 予想の売買に使う価値ゼロ通貨。市場・LMSR・売買・解決はこちらで完結。
2. **賞品ポイント**（新設 prize_wallets）= 景品交換専用。**的中報酬**で付与（予想を当てると貰える）。
3. 現行「ポイント」は **「参加ポイント」に改称**する（Phase Bで実施予定）。
4. 導入は **Phase A→D 段階的**。

### Phase A = 完了済み（0022・テスト全緑・本番適用済み）
- テーブル: `prize_wallets` / `prize_ledger`（FIFO失効用 expires_at 保持）/ `prizes`（景品マスタ）/ `prize_redemptions`（交換申込）
- RPC: `grant_prize_points(p_user,p_amount,p_reason,p_market_id,p_expires_at)`（サーバー専用・有効期限既定90日） /
  `redeem_prize(p_prize_id,p_shipping)`（確定交換・本人・在庫/残高/期限を検証） /
  `expire_prize_points()`（FIFO失効。`max(0, 期限切れ付与合計 − 消費合計)` で正確・二重失効しない）
- RLS: 自分のものだけ読む。prizes は is_active のみ公開。
- 参加pt側は**無改修**。

### 残り（次にやること）
- **Phase B（付与開始）**:
  - `resolve_market`（0005）に的中報酬フックを追加 → 解決時に勝者へ `grant_prize_points` 呼び出し。
    **未決定**: 付与レート（例: 勝ち株1株あたり何賞品pt か）。仁さんに確認してから実装。
  - マイページに賞品pt残高・履歴・有効期限を表示。
  - **現行「ポイント」→「参加ポイント」へUI文言を一括改称**（TopNav/TradePanel/mypage/Footer等。広範囲なので慎重に）。
- **Phase C（景品公開）**:
  - 管理画面に景品マスタCRUD（prizes）と交換申込一覧（prize_redemptions・発送ステータス管理）。
  - 景品一覧ページ＋確定交換UI（配送先入力）。
  - **特商法表記の【要記入】を実会社情報に差し替え**（`web/app/legal/tokushoho/page.tsx` の OPERATOR 定数）。
- **Phase D（運用）**: 日次cronで `expire_prize_points()`（0009/cronに追加）。在庫・配送オペ。

## 6. 法的情報（実装済み・要確認あり）

`/legal/` 配下に4ページ（Footerの「法的情報」からリンク済み）:
- `terms`（利用規約・全14条＋3条の2年齢＋6条景品/景表法遵守明記）
- `privacy`（プライバシーポリシー）
- `tokushoho`（**特定商取引法表記・会社情報は【要記入】プレースホルダ**）
- `no-gambling`（賭博非該当の説明。**断定を避け「〜生じないよう配慮」トーン**に調整済み＝ミライマ流）

賭博非該当の論拠 = **ポイントは無償発行（現金で買えない。提携懸賞/アフィリエイトサイト登録の無料特典＋デイリー付与）→ ユーザーは自己の財産を賭けていない → 賭博罪の「財産上の利益の得喪」を欠く**。景品交換は確定交換（抽選なし）・無償pt利用先。無償発行ゆえ資金決済法の前払式支払手段にも非該当。

### ⚠️ 公開前に必須
1. 特商法表記の【要記入】（運営会社名・責任者・所在地・電話・メール）を実情報へ。
2. **景品設計が固まったら、賭博・景品表示法・特商法の一連を弁護士レビュー**。景品を出す以上ここは外せない（ミライマも景表法遵守を明記）。

## 7. 高負荷対策（実装済み）
