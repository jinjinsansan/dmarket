# 参加ポイント獲得（アフィリエイト成果型）設計案

> ステータス: **提案（未実装）**。作成: 2026-06-20。
> 検討用たたき台。実装前に仁さんの承認＋各ASP/広告主の規約確認を得る。
> 関連: `二層ポイント制_実装記録.md`（参加/賞品ポイントの土台）、README（マネタイズはBETの外側）。

---

## 0. ねらい（ミライマ型）

ユーザーは**金銭を払わず**、提携アフィリエイト案件（例: A8.net等のASP経由の会員登録・資料請求・アプリインストール等）を成果完了すると、運営がアフィリエイト報酬を得て、その原資で**参加ポイントを無償付与**する。

- **賭博非該当を維持**: ユーザーは自己資金を投じない（参加ポイントは無償発行のまま）。
- **マネタイズはBETの外側**: 運営収益＝アフィリエイト成果報酬。
- ✅ **AdSense等のインセンティブ付きクリック広告は採用しない（仁さん決定・2026-06-20）**。アフィリエイト成果型のみで構成（後述§7）。

```
ポイントを貯めるページ → 案件を選ぶ → 計測リンクで広告主サイトへ
  → ユーザーが成果条件を達成（登録等）
  → ASP が成果を確定 → D-market にpostback（S2S通知）
  → 該当ユーザーへ参加ポイントを付与（無償発行）
```

---

## 1. 技術上の最重要ポイント（per-user 成果紐付け）

ASPの成果通知は通常**運営（メディア）単位の集計**で、「どのユーザーが成果したか」は標準では分からない。ポイントサイトを成立させる鍵は:

- **sub-id（任意パラメータ）を計測リンクに埋める**: クリック時に「user_id を含む一意トークン」を sub-id として広告主URLに付与。
- **postback（サーバー間通知 / S2S）で sub-id が返ってくる**: 成果確定時、ASP が sub-id 付きで運営エンドポイントへ通知 → どのユーザーの成果かを特定して付与。

→ **sub-id ＋ postback に対応したASP/案件を選ぶことが前提**。
- 対応が手厚い: ポイントサイト向けを公言するASP（例: afb、アクセストレード、バリューコマース、felmat 等）や成果連携API。
- A8.net は案件により対応がまちまち → **postback非対応の案件は「手動消し込み」運用**（§6 Phase 1）で回す。

## 1.5 実装方式の選択肢：個別ASP直契約 vs オファーウォール内蔵（2026-06-29 追記）

競合 **POYP（ポイプ）の「ためる」タブが、まさにこの参加ポイント獲得＝オファーウォール**で動いていることを確認（`POYP実態調査_2026-06-29.md` §2.5）。実装には2方式あり、**併用が最適**。

| | 個別ASP直契約（afb / アクセストレード） | オファーウォール内蔵（SKYFLAG等） |
|---|---|---|
| 仕組み | 自前で案件選定・計測リンク・sub-id/postback設計 | ASPのWebView/SDKを設置するだけで多数案件を一括掲載 |
| 在庫 | 金融/カード等の**高単価**を制御して選べる | キャリア決済/月額など**大量案件**を即確保 |
| 工数 | 高（自前実装＝本設計のPhase 1〜2） | 低（SDK設置＋成果コールバック受領） |
| per-user紐付け | sub-id＋postback（自前） | ✅ ASP側が標準対応（例: SKYFLAGは `suid`/`spram`(256字)） |

**推奨二層構成**: **SKYFLAG（株式会社Skyfall）を基盤**にオファーウォールで在庫を一気に確保 ＋ **afb/アクセストレード直契約**を高単価案件で併用。
- ASP比較・選定根拠は `POYP実態調査_2026-06-29.md` §5.2 を参照（GMO SmaAD は第二候補だが**現金交換機能は使わない**。maio/メディアレーダー/GMOポイ活はオファーウォールASPではない）。
- ⚠️ **D-marketは換金不可ポイントへの付与に限定**。オファーウォール側の現金/ギフト直接交換導線は採用しない（賭博・資金決済法リスク回避）。
- ⚠️ **賭博法の核心訂正**（`POYP実態調査` §3-3）: 賭博を外すのは「課金経路ゼロ」ではなく「**賭ける対象＝参加ポイントが換金不可で財産的価値ゼロ**」であること。POYPは賭ける対象が換金可のため賭博リスクを直接保有。**D-marketは参加ポイントを換金不可・譲渡禁止に保つ限り安全**（法務再調査 3-0 confirmed）。ただし景表法・特商法・ステマ規制は別途適用。
- ⚠️ 媒体審査リスク: 「予測市場/ベッティング類似」ジャンルがオファーウォール各社の審査で許諾されるか要確認（賭博非該当でも審査NGの可能性）。法人運営が前提。

## 2. データモデル案（新規テーブル）

参加ポイントは既存 `wallets` / `point_ledger` をそのまま使う（無償発行の一種）。新規は計測・成果の層のみ。

```sql
-- 提携案件マスタ（広告主オファー）
create table affiliate_offers (
  id            uuid primary key default gen_random_uuid(),
  name          text not null,            -- 「○○カード発行」等
  description   text,
  image_url     text,
  reward_points bigint not null check (reward_points > 0), -- 成果時に付与する参加pt
  asp           text,                     -- a8 / afb / accesstrade ...
  click_url     text not null,            -- 計測リンク雛形（{TOKEN} を sub-id 差込）
  incentive_ok  boolean not null default false, -- 広告主がインセンティブ付き成果を許可しているか
  hold_days     int not null default 0,   -- 確定保留日数（不正対策）
  is_active     boolean not null default true,
  display_order int not null default 0,
  created_at    timestamptz not null default now()
);

-- クリック（計測トークンの発行記録）
create table affiliate_clicks (
  token       uuid primary key default gen_random_uuid(), -- sub-id として広告主に渡す
  user_id     uuid not null references auth.users(id),
  offer_id    uuid not null references affiliate_offers(id),
  clicked_at  timestamptz not null default now()
);
create index affiliate_clicks_user_idx on affiliate_clicks(user_id, clicked_at);

-- 成果（postback or 手動で作成）
create table affiliate_conversions (
  id           uuid primary key default gen_random_uuid(),
  token        uuid references affiliate_clicks(token), -- どのクリック由来か
  user_id      uuid not null references auth.users(id),
  offer_id     uuid not null references affiliate_offers(id),
  reward_points bigint not null,
  status       text not null default 'pending', -- pending|approved|rejected
  external_id  text,                  -- ASP側の成果ID（冪等キー）
  granted      boolean not null default false, -- 参加pt付与済みか
  created_at   timestamptz not null default now(),
  approved_at  timestamptz,
  unique (offer_id, external_id)       -- 二重計上防止（postback再送対策）
);
```

- **冪等性**: postback は再送され得る。`(offer_id, external_id)` 等で一意制約を張り、二重付与を防ぐ。
- `point_ledger.reason` に **`affiliate`** を追加（0018と同方式で CHECK 制約を貼り替え。現状: signup/daily/buy/sell/redeem/refund/admin_grant/admin_burn）。

## 3. RPC / API

### RPC（SECURITY DEFINER）
| 関数 | 役割 |
|---|---|
| `create_affiliate_click(p_offer_id)` | 本人。トークン発行＋クリック記録 → 計測URL（token差込済み）を返す |
| `grant_affiliate_points(p_user, p_conversion_id)` | **サーバー専用**。成果approved時に参加ptを付与（wallets加算＋point_ledger reason='affiliate'）。granted フラグで冪等 |
| `admin_list_affiliate_*` / `admin_set_conversion_status` | 管理。案件CRUD・成果一覧・承認/却下（承認で付与をトリガー） |

### API ルート
- `GET /api/affiliate/go/[offerId]` … 本人セッションで `create_affiliate_click` → 計測URLへ 302 リダイレクト。
- `POST /api/affiliate/postback` … ASPからのS2S。**共有シークレット/IP許可で検証**。token・external_id・status を受け、`affiliate_conversions` を upsert。`approved` なら `grant_affiliate_points` を service_role で実行。

## 4. ユーザー導線

- 「**ポイントを貯める**」ページ（新規・`/earn` 等）: 有効な `affiliate_offers` を一覧（案件名・付与pt・説明）→「参加する」で `/api/affiliate/go/[id]` へ。
- マイページ: 取得履歴（point_ledger reason='affiliate'）＋保留中の成果（pending）を表示。
- TopNav/BottomNav に導線（「貯める」）。

---

## 5. 不正検知（重要・既存 anti-fraud 0013 の思想を拡張）

- 多重アカウント（同一端末/IP/LINEの重複）での成果反復をブロック。
- 自己成果・トークン使い回し・postback 偽装（シークレット必須）対策。
- `hold_days` による確定保留（広告主の承認待ち＝多くの案件で成果は後日確定/却下される）。
- 高額案件は手動レビュー。付与は**承認後のみ**。

## 6. 段階導入

1. **Phase 1（手動消し込み・postback不要で即始められる）= 実装済み（0026・ローカルテスト全緑・ビルドOK・本番SQL未適用）**: 案件マスタ＋クリック計測＋管理画面で「ASPの成果レポートの sub-id(token) を突き合わせて承認→付与」。A8.net等postback非対応でも回る。
   - DB 0026: `affiliate_offers`/`affiliate_clicks`/`affiliate_conversions`＋RPC（`create_affiliate_click`・`admin_approve_conversion`・`admin_list_offers`/`upsert_offer`/`set_offer_active`・`admin_list_conversions`・`admin_recent_clicks`）。`point_ledger.reason` に `affiliate` 追加。absence test に `admin_approve_conversion` を無償発行として登録。
   - UI: ユーザー `/earn`（案件一覧→「貯める」で計測リンクへ遷移）。管理 `/admin/affiliate`（案件CRUD＋token承認＋消し込み待ち/履歴）。TopNav/BottomNav に「貯める」、管理ダッシュボードに「提携案件」。
   - テスト `0010_affiliate.sql`。
2. **Phase 2（自動化）**: sub-id＋postback対応ASPで `/api/affiliate/postback` を有効化 → 自動付与。
3. **Phase 3（堅牢化）**: 不正検知強化・保留/却下フロー・案件ごとの上限・レート調整（platform_settings）。

## 7. 法務・規約（実装前に必須確認）

- ✅ **無償発行は維持**（ユーザーは金銭を払わない）→ 賭博非該当・資金決済法非該当の論拠はそのまま。既存 legal文面が「提携サイトの無料特典」を既に明記。
- ⚠️ **Google AdSense はインセンティブ付きクリック（クリック/フォーム入力の見返りに報酬・ポイント付与）を禁止**。規約違反でアカウント停止リスク大 → **採用しない**。広告型をやるならインセンティブ許可の別ネットワークのみ。
- ⚠️ **アフィリエイト広告主ごとに「インセンティブ付き成果（ポイント付与目的の登録）」の可否規約が異なる**。`incentive_ok=true` の案件のみ掲載する運用に。
- プライバシー: クリック/成果はユーザーに紐づく → プライバシーポリシーに追記（現状も「提携サイト連携情報」を記載済み、範囲を具体化）。
- 景表法/特商法: ポイントは無償・運営収益はアフィリエイト報酬という整理を維持。景品設計と合わせ**公開前に弁護士レビュー**（既定方針）。

## 8. 判断ポイント（仁さんへ）

- ✅ **AdSense系のクリック報酬は不採用**（決定済み・2026-06-20）。アフィリエイト成果型のみ。

残る判断:
1. **使うASP**: postback/sub-id 対応を優先するか（自動化）、まずA8.net中心で**手動消し込み（Phase 1）**から始めるか。
2. **付与レート**: 案件ごとの `reward_points`（アフィリエイト報酬額に対して何ptを還元するか＝原資の何%をユーザーに渡すか）。
3. **保留/承認ポリシー**: 即時付与 vs 承認後付与（不正対策上は承認後を推奨）。

→ 上記が決まれば、Phase 1（案件マスタ＋クリック計測＋管理消し込み）から着手できる。既存の管理画面・RPC・台帳パターンに載るため、追加は計測/成果の一層のみ。
