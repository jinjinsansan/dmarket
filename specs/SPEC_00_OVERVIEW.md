# SPEC-00: 全体概要・法的フレーミング・アーキテクチャ

ポイント制予測市場プラットフォーム（換金不可・賭博非該当）の**マスタードキュメント**。
全SPECの索引、設計を貫く法的・技術的前提、システム全体像、用語集を定義する。

> 免責: 本書の法的記述は設計上の制約条件を整理したものであり、法的助言ではない。
> 公開前に賭博罪・景品表示法・資金決済法に詳しい専門家のレビューを必ず受けること。

---

## 1. プロダクト要旨

実世界の事象（競馬・FX・クリプト・ニュース等）の結果を予測し、**換金不可のポイント**で売買する
予測市場。価格＝確率がトレードで動く Polymarket 型の体験を提供するが、**一切の資産BETを行わない遊び場**。
報酬はランキングと称号のみ（賞品ゼロ）。マネタイズはBETの外側（コスメ課金・広告・有料機能）で行う。

---

## 2. 法的フレーミング（= エンジニアリング制約）

賭博罪の構成要件は「偶然の勝敗」「財産的価値のあるものを賭ける」「得喪を争う」。
本プラットフォームは**財産的価値をゼロにする**ことで賭博を構成しない設計とする。
以下を**システム的に強制される不変条件**として全SPECが守る。

1. **換金不可** — ポイントを現金・暗号資産・物品に交換するRPC/導線を一切作らない。
2. **有償発行禁止** — ポイントを現金で購入する導線を作らない。発行は無償のみ（登録・デイリー・的中償還）。
3. **譲渡禁止** — ユーザー間のポイント送付・取引・贈与を作らない。
4. **賞品ゼロ（v1）** — 上位者への金券・物品の付与を行わない。報酬は称号・ランキング表示のみ。
5. **二次流通の遮断** — ポイント・称号・NFT等に二次市場価値が付く仕組みを作らない。
6. **マネタイズの分離** — 課金はアバター/テーマ/有料分析/広告など**BETに使えない領域**に限定。
   有償で得たものをBET原資に変換する導線は禁止（SPEC-08で強制）。

これらは単なる方針ではなく、コードに存在してはならない機能のリストである。
「存在しないこと」をテストで担保する（各SPECの受け入れ条件）。

## 6. 市場のライフサイクル（端から端まで）

```
1. 供給(04): 管理者作成 / 自前テンプレ / Polyミラー で market+outcomes を生成（初期q）
2. 取引(02): ユーザーが buy/sell。LMSRで価格=確率が動く。台帳に全移動を記録。Realtimeで配信(05)
3. 締切: close_time で取引停止 (open→closed)
4. 解決(03): 自動フィード or 管理者(07) が勝ちoutcomeを確定 → resolve_market(02)
5. 償還(02): 勝ち株×100ptを一括付与。負け株は0
6. 反映(05/06): ポートフォリオに損益、リーダーボードにスコア反映
```

## 7. グローバル用語集

- **市場 (market)**: 1つの予測対象。binary(YES/NO) または multi。
- **アウトカム (outcome)**: 市場の選択肢。`q`（累積発行株数）がLMSR状態。
- **株 (share)**: アウトカムの持ち分。勝てば `POINTS_PER_SHARE(=100)` pt、負ければ0で償還。
- **価格 (price)**: アウトカムの確率 (0..1)。¢表記は ×100。
- **b (流動性)**: LMSRの価格の動きにくさ。市場ごとに保持。
- **ポイント (point)**: 換金不可の遊技内通貨。`wallets.balance`、移動は `point_ledger`。
- **供給源 (source)**: `admin` / `template` / `mirror`。
- **解決方式 (resolution_kind)**: `manual` / `auto`。
- **gap**: カテゴリで目標数に足りずPolyミラーで埋める数。

## 8. クロスカット不変条件（全SPEC共通）

1. 残高 = 当該ユーザーの台帳 delta 総和（常時一致・監査可能）。
2. ポイントの入手経路は無償3種のみ。換金・購入・譲渡のRPCは存在しない。
3. 残高・株・q の書き込みは security definer RPC 経由のみ（クライアント直書き禁止／RLS）。
4. 価格を読む取引は対象 market を行ロックしてから計算・更新（レース条件防止）。
5. 終端状態（resolved/void）は不可逆。
6. 端数はシステム有利側（買い切上げ・売り切捨て）でポイント総量が膨張しない。
7. 1人1ウォレット（SPEC-01で本人性を担保）。

## 9. 環境・デプロイ（概要）

- Supabase プロジェクト（本番/ステージング分離）。マイグレーションでスキーマ管理。
- Edge Functions に外部APIキー（価格・スポーツ等）を環境変数で保持。Polyは鍵不要。
- pg_cron で供給(15分)・解決(5分)・リーダーボード集計(SPEC-06)をスケジュール。
- フロントは Vercel 等。Realtime は Supabase。
