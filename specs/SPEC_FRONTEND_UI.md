# SPEC: フロントエンド (Next.js / Polymarket-style UI)

予測市場プラットフォームのWeb UI。**Polymarketのレイアウト・UX・コンポーネント挙動を忠実に踏襲し、
ブランド要素（名前・ロゴ・配色・フォント）のみ自前**とする。レスポンシブWeb（PC/スマホ両対応）。

> 依存: `SPEC_CORE_PRICING_LEDGER.md`（RPC・テーブル・Realtime）/ `SPEC_MARKET_SUPPLY_LAYER.md`（市場・カテゴリ）。
> 本SPECは中核のRPC名・戻り値形・Realtimeチャネルをそのまま利用する。

---

## 0. スタックと依存

- Next.js (App Router) + TypeScript
- Supabase JS client（`@supabase/supabase-js` / `@supabase/ssr`）— 認証・データ取得・Realtime
- Tailwind CSS（デザイントークンはCSS変数で定義しブランド差し替え可能に）
- チャート: **Recharts**（Area/Lineで確率推移）を既定。TradingView風の手触りが欲しければ `lightweight-charts` に差し替え可
- 状態: Server Componentsで一覧・詳細の初期取得、トレードパネルとRealtimeはClient Component

**重要原則**: 価格計算・残高更新は一切クライアントで確定しない。クライアントのLMSR計算は
**注文プレビュー（見積もり表示）専用**で、確定値は必ず `buy_shares`/`sell_shares` RPCの戻り値を使う（§8）。

## 1. 中核SPECへの追記（価格履歴）— 先に実装

確率チャート（v1必須）に履歴が要る。中核の取引RPCに以下を追記する。

```sql
create table market_price_history (
  id          bigint generated always as identity primary key,
  market_id   uuid not null references markets(id),
  outcome_id  uuid not null references outcomes(id),
  price       numeric not null,            -- 取引直後の確率 (0..1)
  recorded_at timestamptz not null default now()
);
-- index: (market_id, recorded_at)

-- buy_shares / sell_shares の q 更新後に、その市場の全outcomeのprice点を1行ずつINSERT。
-- 加えて市場生成時（初期q）にも初期点を1点入れる。
-- RLS: SELECT public (true) / 書き込みはRPC経由のみ。
```

これにより各市場の確率推移が時系列で残り、チャートで描ける。

## 2. デザイントークン（ブランドは自前・機能色はPolymarket慣習）

CSS変数で定義し、`--brand-*` を差し替えるだけでブランド変更可能にする。

```css
:root {
  /* ブランド（自前に差し替え。以下は中立プレースホルダ） */
  --brand-primary:   #2D6CDF;   /* CTA・リンク・アクティブ */
  --brand-bg:        #0E1117;   /* 既定はダーク基調（ライト版も別トークンで用意） */
  --brand-surface:   #161B22;   /* カード面 */
  --brand-border:    #232A34;
  --brand-text:      #E6EAF0;
  --brand-text-dim:  #8A93A2;

  /* 機能色（Polymarket慣習＝予測市場の定石。これは踏襲して可） */
  --pos:  #1FA971;   /* YES / 上昇 / 利益（緑） */
  --neg:  #E5484D;   /* NO  / 下降 / 損失（赤） */

  /* タイポ（自前。data用に等幅を1つ） */
  --font-display: /* 見出し用の自前フォント */;
  --font-body:    /* 本文 */;
  --font-mono:    /* 価格・確率・数値（等幅） */;

  /* レイアウト */
  --radius: 10px;  --radius-sm: 6px;
  --gap: 16px;
}
```

確率・価格・残高などの**数値は必ず等幅フォント**で表示（桁が揺れず読みやすい＝取引UIの定石）。

## 4. 画面A: 市場一覧 (Markets)

Polymarketのカードグリッドを踏襲。3画面の1つとして作り込む。

```
┌─ MarketCard ───────────────────────────┐
│ [カテゴリ] [画像/アイコン]               │
│ 質問テキスト（2行までで省略）            │
│                                         │
│ ┌─ 確率バー ─────────────────────────┐  │  ← 二択: YES%を緑、残りを赤の横バー
│ │██████████ 64%        NO 36%        │  │
│ └────────────────────────────────────┘  │
│ 出来高/参加  ·  締切まで 残り2日          │
│ [ YES 64¢ ]              [ NO 36¢ ]      │  ← カード上から直接ベット可（Polymarket流）
└─────────────────────────────────────────┘
```

- **カード構成**: カテゴリラベル、質問、確率バー（二択はYES/NO、多択は上位outcomeを積み上げ）、
  メタ情報（参加人数・締切）、YES/NOクイックベットボタン（価格を ¢ 表記＝確率×100）。
- **カテゴリタブ**でフィルタ、**検索**で質問文の全文フィルタ。
- **ソート**: 新着 / 締切が近い / 参加が多い。
- カード価格は確率の Realtime 更新を受けて滑らかに変わる（§7）。
- グリッドは PC 3〜4列 / タブレット 2列 / スマホ 1列。

## 5. 画面B: 市場詳細 + トレードパネル（最重要・心臓部）

Polymarketの「左に情報・チャート、右にトレードパネル」構成を踏襲。

```
┌─ 市場詳細 (PC: 2カラム) ───────────────────────────────────┐
│ [カテゴリ] 質問テキスト（大見出し）                          │
│ 現在: YES 64%  /  NO 36%   締切 6/19 15:00  状態: 取引中      │
│                                                              │
│ ┌─ 確率チャート ─────────────────┐  ┌─ トレードパネル ───┐ │
│ │   ╱╲      ╱                     │  │ [ 買う | 売る ]      │ │
│ │  ╱  ╲___╱     YES 推移          │  │ [ YES ] [ NO ]      │ │
│ │ ────────────────── 時間 →       │  │ 数量: [   10  ] 株  │ │
│ │ [1H][6H][1D][1W][ALL]           │  │ ─────────────────── │ │
│ └─────────────────────────────────┘  │ 予想コスト: 640 pt   │ │
│                                       │ 約定後YES: 64%→67%   │ │
│ ▸ 説明 / 解決ルール                    │ 保有後: 20株          │ │
│ ▸ 解決の根拠（resolutions: ソースURL） │ [   YESを買う   ]    │ │
│ ▸ あなたの保有: YES 10株 (取得 600pt)  └──────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

**トレードパネルの挙動（Polymarket流）**
- 上部トグル: **買う / 売る**。その下に **YES / NO**（多択なら outcome チップ）。
- 数量入力（株数）。入力に応じて**クライアントLMSRで即時プレビュー**（§8）:
  - 予想コスト（買い）または予想受取（売り）を等幅で表示
  - 約定後の確率変化（64%→67% のように矢印表示）
  - 売りのときは保有株数を上限にバリデーション
- 「YESを買う」ボタン押下 → `buy_shares(outcome_id, shares)` を呼ぶ。
  - 成功: 戻り値の `cost_points`・`new_prices`・`balance` で確定表示し、トースト「YESを10株購入」。
  - 失敗: `insufficient_balance`→「ポイントが足りません」、`market_closed`→「この市場は締め切られました」。
- **状態が `open` 以外**ならパネルは入力不可にし、状態に応じた説明を出す
  （closed=「解決待ち」、resolved=結果と自分の損益、void=「中止・返金済み」）。
- 解決済み市場では、トレードパネルの位置に**結果カード**（勝ちoutcome・自分の償還額）を表示。

**解決の透明性**: `resolutions` の `source_url`・`resolved_at`・`resolution_kind` を必ず表示（信頼の核）。

## 6. 画面C: ポートフォリオ (Portfolio)

3画面の1つ。中核の `positions`（cost_basis）と `point_ledger` を使う。

```
┌─ サマリ ──────────────────────────────────────────┐
│ 残高 1,250 pt   保有評価 +320 pt   通算損益 +1,050 pt │
├─ アクティブ保有 ──────────────────────────────────┤
│ 市場 / outcome / 保有株 / 取得 / 現在値 / 含み損益    │
│ …各行クリックで市場詳細へ                            │
├─ 解決済み ────────────────────────────────────────┤
│ 市場 / 結果 / 償還 / 確定損益                         │
├─ 取引履歴（台帳） ────────────────────────────────┤
│ 日時 / 種別(付与/買/売/償還/返金) / 増減 / 残高        │
└───────────────────────────────────────────────────┘
```

- **含み損益** = 現在価格で売却したときの受取（クライアントLMSRで概算）− `cost_basis`。
- **確定損益** = 償還額 − `cost_basis`（解決済み）。
- 取引履歴は `point_ledger` をそのまま時系列表示（`reason` を日本語ラベルに）。
- 数値は緑（+）/赤（−）で色分け（機能色 `--pos`/`--neg`）。

## 7. Realtime 連携

- 表示中の市場について `outcomes` テーブルの `q` 変更を `market_id` フィルタで購読。
- `q` 更新を受けたら、その市場の全 outcome の価格を `lmsr_price()`（クライアント版）で再計算し、
  確率バー・¢表記・チャートの最新点を更新。
- 一覧画面では、画面内に見えているカードの市場だけ購読（パフォーマンス）。

## 8. クライアントLMSR（プレビュー専用）

`SPEC_CORE_PRICING_LEDGER.md` §2 と**同一の数式**をTypeScriptで実装（数値安定版・log-sum-exp）。

```ts
function lmsrCost(q: number[], b: number): number {
  const m = Math.max(...q.map(x => x / b));
  return b * (m + Math.log(q.reduce((s, x) => s + Math.exp(x / b - m), 0)));
}
function lmsrPrice(q: number[], b: number, k: number): number {
  const m = Math.max(...q.map(x => x / b));
  const e = q.map(x => Math.exp(x / b - m));
  return e[k] / e.reduce((s, x) => s + x, 0);
}
// 注文プレビュー: コスト見積り・約定後価格
//   buyCostPreview  = ceil((lmsrCost(qWithDelta) - lmsrCost(q)) * POINTS_PER_SHARE)
//   sellRecvPreview = floor((lmsrCost(q) - lmsrCost(qMinusDelta)) * POINTS_PER_SHARE)
```

**確定値は必ずRPCの戻り値**。プレビューと実約定が端数でズレうる旨は、UIで「予想」と明示。

## 9. UXライティング（文言の作法）

- **能動態・命令形**で、起きることをそのまま書く: 「YESを買う」「ボーナスを受け取る」。
- ボタンの語をフロー全体で一貫させる: 「購入する」を押したトーストは「購入しました」。
- **ユーザーが操作・認識する言葉で命名**し、システム内部用語を出さない（"outcome" ではなく選択肢名）。
- **エラーは謝らず、何が起きどう直すかを言う**: 「ポイントが足りません。デイリーボーナスを受け取れます」。
- **空状態は行動への誘い**: 保有ゼロなら「まだ予想がありません。市場を見る →」。
- 数値・確率・損益は等幅・色分けで一目で読めるように。

## 10. レスポンシブ

- ブレークポイント: モバイル <768、タブレット 768–1024、PC >1024。
- 市場詳細: PCは2カラム（左チャート/右トレードパネル）。**モバイルはトレードパネルを
  下部のボトムシート**（「ベットする」ボタンで迫り上がる）にして、チャートを全幅に。
- 一覧グリッドは §4 の列数で可変。
- タップ標的は最低44px、キーボードフォーカス可視、`prefers-reduced-motion` 尊重。

---

## 11. 状態（空・読み込み・エラー）

- 読み込み: カード/チャート/パネルはスケルトン表示。
- 空: カテゴリに市場ゼロ→「このカテゴリはまだ市場がありません」＋他カテゴリ誘導。
- エラー: 取得失敗は再試行ボタン付きの簡潔メッセージ（謝罪文にしない）。
- RPC失敗は §5 の各 reason を日本語化してトースト。

---

## 12. 受け入れ条件 (Acceptance Criteria)

- [ ] 一覧カードのYES/NO価格が、他ユーザーのトレードでRealtimeに更新される。
- [ ] 市場詳細で株数を入れると、予想コストと約定後確率がクライアントLMSRで即時表示される。
- [ ] 「YESを買う」で `buy_shares` が呼ばれ、戻り値の確定コスト・残高・新価格でUIが更新される。
- [ ] 残高不足/締切後の購入が、それぞれ適切な日本語メッセージで弾かれる。
- [ ] 確率チャートが `market_price_history` から時系列で描画され、期間切替（1H/1D/1W/ALL）が効く。
- [ ] 解決済み市場では、トレードパネルが結果カード（勝ちoutcome＋自分の償還）に置き換わる。
- [ ] 解決の根拠（source_url・確定日時）が市場詳細に表示される。
- [ ] ポートフォリオで含み損益・確定損益・取引履歴が正しく表示される。
- [ ] デイリーボーナスを受け取ると残高が増え、同日は非活性になる。
- [ ] モバイルでトレードパネルがボトムシートとして機能する。
- [ ] ブランドトークン（`--brand-*`）を差し替えるだけで全体の見た目が変わる。

## 13. 実装順序の推奨

1. デザイントークン（§2）＋グローバルレイアウト/ナビ（§3）＋残高表示・デイリー受領。
2. 中核への価格履歴追記（§1）。チャートのデータ源を先に確保。
3. クライアントLMSR（§8）— 一覧・詳細・ポートフォリオの全価格表示が依存。
4. 市場一覧（§4）＋Realtime購読（§7）。
5. 市場詳細＋トレードパネル（§5）— buy/sell RPC接続。**ここでベットが回る。**
6. 確率チャート（§5右上、Recharts）。
7. ポートフォリオ（§6）。
8. レスポンシブ/ボトムシート（§10）、空・エラー状態（§11）、受け入れ条件（§12）検証。
